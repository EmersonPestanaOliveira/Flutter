Case Tecnico Gabriel: Mapa Comunitario

Roteiro de apresentacao tecnica com arquitetura, decisoes, analogias e fluxos.

Abertura sugerida: Eu pensei o Mapa Comunitario como duas funcionalidades que precisam conversar muito bem: primeiro, o usuario precisa conseguir registrar uma ocorrencia mesmo sem internet; segundo, o mapa precisa continuar fluido mesmo com muitos pins. Por isso a arquitetura gira em torno de tres ideias: offline-first, carregamento por viewport e observabilidade.
Usuario -> App Flutter -> Fila local SQLite -> Sync Worker -> Firestore + Storage \-> Mapa Comunitario <- Alertas remotos + pendentes locais

1. Offline-First

O ponto principal e que a UI nao depende da rede para confirmar o relato. Quando o usuario envia uma ocorrencia, o app gera um clientId, salva o payload no SQLite, persiste os anexos e retorna sucesso local. O envio real acontece depois, pelo sync.

Formulario -> UseCase -> Repository -> SQLite Outbox -> sucesso local para UI SyncWorker -> busca queued/failed -> Firestore + Storage -> markSynced -> cleanup

Por que SQLite?

Para uma fila local pequena e critica, SQL explicito e simples, auditavel e facil de testar em memoria. Nao ha necessidade de uma camada maior com codegen para esse volume e esse tipo de dado.

Por que clientId?

Para idempotencia. O clientId funciona como numero de protocolo: se o sync tentar enviar duas vezes, ele grava no mesmo documento remoto em vez de duplicar a ocorrencia.

Analogia: o app e como uma agencia dos Correios. Mesmo sem internet, o usuario entrega a carta. Ela fica numa caixa local segura. Quando a conexao volta, um funcionario tenta enviar. Se falhar, agenda nova tentativa; se falhar muitas vezes, separa para revisao manual.

2. Seguranca Local

O payload textual e numerico da ocorrencia e criptografado antes de ir para o SQLite. A chave fica no secure storage. Os anexos ficam em armazenamento persistente proprio, para sobreviver a restart do app.

  • Payload textual: SQLite, criptografado.
  • Anexos: arquivos persistidos localmente antes do upload.
  • Upload: Firebase Storage no momento do sync.
  • Documento final: Firestore usando clientId como ID.

3. Sync Resiliente

O sync tem tres gatilhos: tentativa imediata quando o app ja esta online, WorkManager periodico e orquestrador quando a rede volta.

SQLite queued/failed | v SyncWorker -> tenta enviar |-- sucesso ----------> markSynced + limpa anexos |-- falha recuperavel -> backoff + retry |-- muitas falhas ----> deadLetter -> retry manual

Por que backoff com jitter?

Para evitar que muitos usuarios batam no backend ao mesmo tempo quando a rede volta. Cada cliente espera um intervalo ligeiramente diferente.

Analogia: se uma central telefonica cai e volta, voce nao quer todo mundo ligando no mesmo segundo. O backoff distribui essas tentativas.

4. Mapa Com Muitos Pins

O maior erro seria buscar todos os pins sempre. A solucao usa carregamento por viewport: o app consulta apenas os alertas dentro da area visivel do mapa, com um buffer ao redor.

Usuario move o mapa | v onCameraIdle -> debounce 300ms -> calcula viewport + buffer | v Viewport ja coberto? |-- sim: reusa pins carregados |-- nao: query por GeoHash -> mescla pins -> renderiza

Por que onCameraIdle?

Durante o gesto, o mapa pode emitir muitos eventos. A app so precisa agir quando o usuario para de mexer, reduzindo chamadas e processamento.

Por que buffer?

Se o usuario arrasta pouco o mapa, a nova area visivel ainda esta dentro da area ja carregada. Isso evita flicker e reduz chamadas ao backend.

Analogia: e como streaming que baixa o trecho atual e um pouco do proximo. Quando voce avanca um pouco, nao precisa carregar tudo de novo.

5. GeoHash

GeoHash transforma latitude e longitude em uma string ordenavel. Isso permite consultar ranges no Firestore para cobrir o viewport.

Viewport -> celulas GeoHash -> ranges lower/upper -> queries Firestore -> filtro preciso em memoria -> pins finais

Mesmo usando GeoHash, existe filtro local defensivo porque a celula cobre uma area aproximada. A query reduz muito o conjunto de dados; o filtro local garante precisao nas bordas.

6. Clustering

Renderizar muitos markers individuais no Google Maps custa caro. O clustering agrupa pins quando o volume e alto e o zoom esta mais aberto.

Lista de alertas -> precisa clusterizar? |-- poucos pins ou zoom alto -> pins individuais |-- muitos pins -> agrupa por celula -> marker de cluster

Cluster nao deve ser usado sempre. No zoom alto, o usuario quer precisao; em visao macro, ele quer densidade.

Analogia: olhando o Brasil, faz sentido ver "200 ocorrencias em Sao Paulo". Olhando uma rua, o usuario quer o pin exato.

7. Cache de Icones

Criar BitmapDescriptor para marker tem custo. Por isso os icones sao cacheados por tipo de alerta e por bucket de cluster.

AlertaTipo -> PinCache tem? |-- sim -> retorna BitmapDescriptor |-- nao -> gera icone -> salva no cache -> retorna

8. Estado e Arquitetura

A arquitetura segue uma separacao em camadas: Presentation, Domain, Data e Core.

Presentation: Screens + Cubits | v Domain: Entities + UseCases + Repository contracts | v Data: Firestore / SQLite / Storage implementations | v Core: DI, sync, network, telemetry, security

Por que Cubit?

O estado e complexo, mas as acoes sao diretas: carregar dados, aplicar filtro, mover mapa, selecionar pin e tentar retry. Cubit da previsibilidade sem criar muita cerimonia com eventos.

Por que UseCase?

Para a UI nao conhecer Firestore, SQLite, Storage ou detalhes de sync. A tela chama uma intencao de negocio, e a implementacao fica atras das interfaces.

9. Filtros

Filtro de tipo e data entra no estado do mapa. Quando muda, o cache e invalidado e uma nova query e feita no viewport atual.

Usuario seleciona filtro -> atualiza AlertaFilter -> limpa acumulador -> refaz query no viewport -> mapa atualizado

O ideal e empurrar o filtro para o Firestore. Quando isso nao e possivel, o app mantem filtro local defensivo.

10. Observabilidade

A solucao foi pensada para diagnosticar producao. O objetivo nao e apenas funcionar no aparelho local, mas saber quando degradou e por que.

  • Mapa: pins carregados, tempo de query, fallback, cluster, cache hit/miss.
  • Sync: total, sucesso, falha, dead-letter, tempo em fila.
  • Offline: tamanho da outbox, idade media/maxima, quantidade de retries.
  • Erros: Crashlytics e Performance atras de uma API mockavel.
Frase boa para entrevista: eu nao quero so que funcione no meu aparelho. Eu quero saber quando degradou em producao e por que.

11. Testabilidade

A estrategia de testes foca na logica que da para isolar:

  • DAO SQLite em memoria.
  • Sync worker, backoff e dead-letter.
  • Filtros, cluster, cache de pins e geo utils.
  • Cubits e benchmark de viewport.

O Google Maps em si fica melhor para teste de integracao/plataforma. Em unit test, o valor esta na logica que decide quais markers mostrar.

Resumo Final

Minha proposta e um app offline-first com outbox local criptografada, sync idempotente por clientId, retry com backoff e dead-letter. Para o mapa, evito buscar tudo: carrego por viewport com GeoHash, aplico filtros no backend quando possivel, mantenho filtro defensivo local, uso buffer para reduzir chamadas, clustering para preservar fluidez e cache de icones para reduzir custo de renderizacao. A arquitetura separa UI, dominio e dados, e a observabilidade foi pensada para diagnosticar gargalos reais.

Pontos Para Assumir Com Honestidade

Hoje eu vejo tres melhorias antes de producao plena: consolidar a tela real do formulario com o Cubit testado, tornar o caminho GeoHash obrigatorio para bases grandes em vez de depender do fallback limitado, e configurar o background sync nativo no iOS. Sao ajustes de maturidade, nao mudancas de arquitetura.
